\title{Formalizing a Price-Time Priority Matching Engine in TLA+:\\
A Specification Case Study and Guided Implementation Synthesis}

\author{}
\date{}

\begin{document}

\maketitle

\begin{abstract}
We report on the formalization and bounded exhaustive exploration of a
nontrivial financial matching-engine specification using TLA+ and the TLC
model checker. The specification covers a full order-type suite---LIMIT,
MARKET, MARKET-TO-LIMIT, STOP\_LIMIT, STOP\_MARKET---augmented with iceberg
orders, four self-trade prevention (STP) policies, post-only, Fill-Or-Kill,
minimum-quantity, cancel, and amend semantics. Within the explored
configurations TLC found one definitive FIFO invariant violation caused by a
missing timestamp refresh on triggered stop orders, and surfaced a second
specification gap---an iceberg order stranding reachable when STP DECREMENT
zeros a resting order's visible quantity without triggering the iceberg reload
path. 

Crucially, we extend this case study to the implementation phase, demonstrating how 
a rigorous formal specification enables ``one-shot'' synthesis of high-performance 
C++ and C matching engines via large language models (LLMs). By providing the 
formal specification as an unambiguous ``logical prompt'' alongside high-level 
architectural hints (intrusive lists and TLSF allocation), we achieved logically 
correct, production-grade implementations that pass all formal conformance tests 
on the first attempt. This paper describes the model, the invariants, the 
counterexamples, the fixes, and the emerging paradigm of ``Guided Synthesis'' 
for critical financial infrastructure.
\end{abstract}

\section{Introduction}
% ============================================================================

Matching-engine correctness is commercially important and surprisingly
difficult to state precisely. Production engines combine a large number of
interacting features---order types, time-in-force policies, self-trade
prevention, hidden quantity, stop triggers, cascading fills---each specified
correctly in isolation, but capable of producing unexpected emergent behavior
when combined. Prose specifications resolve the easy cases and leave the hard
ones to implementor judgment.

Formal methods offer a sharper alternative: write the semantics as an
executable state machine and enumerate its reachable states. This paper is an
experience report on doing exactly that for a matching engine specification. 
Furthermore, we explore the ``Implementation Gap''---the distance between a 
verified specification and a high-performance implementation. We show that a 
formal specification, when used as the primary source of truth for an AI 
assistant, can drive the generation of complex, optimized code that is 
``correct-by-construction'' relative to the formal model.

The contributions of this paper are:
\begin{enumerate}[nosep]
\item A TLA+ formalization of a feature-rich price-time-priority matching
      engine specification.
\item A suite of invariants covering book consistency, queue fairness, and
      order-type safety conditions.
\item A concrete TLC counterexample showing that triggered stop orders can
      violate FIFO if activation does not refresh queue priority.
\item Identification of an additional semantic flaw involving self-trade-prevention
      decrement and iceberg visibility management.
\item Demonstration of \textbf{Guided Synthesis}: using the formal specification 
      to generate logically correct C++ and C implementations in a single pass.
\item Minimal specification amendments and lessons about composing advanced
      order semantics.
\end{enumerate}

The paper is organized as follows. Section~\ref{sec:semantics} describes the
matching-engine semantics under study. Section~\ref{sec:model} presents the
TLA+ model and the scope of the bounded exploration.
Section~\ref{sec:invariants} enumerates the invariants checked.
Section~\ref{sec:findings} presents the findings in detail, with
counterexample traces. Section~\ref{sec:fixes} shows the fixes applied.
Section~\ref{sec:synthesis} discusses the implementation via Guided Synthesis.
Section~\ref{sec:lessons} draws lessons about feature-composition hazards.
Section~\ref{sec:limitations} states the limitations and reproducibility
conditions. Section~\ref{sec:conclusion} concludes.

\section{Matching-Engine Semantics Under Study}
\label{sec:semantics}
% ============================================================================

\subsection{Specification Source}

The object of study is an internal prose specification,
\texttt{matching-engine-formal-spec.md} version~1.2.0, describing a generic
asset-agnostic order matching engine with price-time priority (FIFO). The
specification defines primitive domains, order well-formedness constraints, the
order book structure, a core matching algorithm, post-match disposition, six
order types, four STP policies, iceberg reload semantics, stop triggering with
cascading evaluation, cancel, and amend.

The specification is 900 lines of structured pseudocode. It was written with
care---well-formedness constraints are enumerated, invariants are stated
formally, and edge cases such as MTL price derivation and FOK dry-run
semantics are treated explicitly. It is not a toy specification.


\subsection{Order Types and Features}

The engine supports five order types:
\begin{itemize}[nosep]
\item \textbf{LIMIT}: rests at a specified price; fills at that price or better.
\item \textbf{MARKET}: fills at any available price; never rests on the book.
\item \textbf{MARKET-TO-LIMIT (MTL)}: aggressively matches at any price; locks
      to the first execution price and rests any remainder as a LIMIT order.
\item \textbf{STOP\_LIMIT / STOP\_MARKET}: dormant until a trigger condition on
      the last trade price is satisfied, then converted to LIMIT or MARKET
      respectively and re-entered into the matching pipeline.
\end{itemize}

Order modifiers include:
\begin{itemize}[nosep]
\item \textbf{Iceberg} (\texttt{displayQty}): only
      $\mathit{visibleQty} \leq \mathit{displayQty}$ is shown to the market;
      after the visible slice is consumed, the order reloads with a fresh
      visible slice and loses time priority.
\item \textbf{Post-Only}: the order must not aggress; rejected (under the
      REJECT policy) if it would cross the spread.
\item \textbf{TimeInForce}: GTC, GTD, IOC, FOK, DAY.
\item \textbf{MinQty}: requires a minimum fill on first execution; the
      constraint is cleared after the initial threshold is met, allowing the
      remainder to rest.
\item \textbf{Self-Trade Prevention (STP)}: four policies---CANCEL\_NEWEST,
      CANCEL\_OLDEST, CANCEL\_BOTH, DECREMENT---govern what happens when an
      incoming order would trade against a resting order in the same STP group.
      The incoming order's policy governs.
\end{itemize}


\subsection{Core Invariants Stated in the Specification}

The prose specification states fourteen invariants that must hold after every
\texttt{PROCESS} call completes (\S13 of the spec). The FIFO invariant
(INV-7/INV-8 in the spec) is particularly relevant here:

\begin{quote}
For every price level, consecutive orders satisfy:
$o_1.\mathit{timestamp} < o_2.\mathit{timestamp}$.
\end{quote}

This is the formal statement of price-time priority. A violation means an
order with a later arrival time has higher queue position than one that arrived
earlier---the queue is out of order.

\section{TLA+ Model and Bounded Exploration}
\label{sec:model}
% ============================================================================

\subsection{Model Structure}

The TLA+ specification (\texttt{MatchingEngine.tla}, approximately 850 lines)
is a direct translation of the prose specification. The state consists of
eight variables:

\begin{lstlisting}[language=tlaplus]
bidQ           : PRICES -> Seq(Order)   (* per-price bid queues *)
askQ           : PRICES -> Seq(Order)   (* per-price ask queues *)
stops          : Set(Order)             (* dormant stop orders *)
lastTradePrice : PRICES \cup {NULL}     (* last execution price *)
postOnlyOK     : Bool                   (* no post-only aggressor trade *)
stpOK          : Bool                   (* no same-group trade *)
nextId         : Nat                    (* next order ID *)
clock          : Nat                    (* logical clock *)
\end{lstlisting}

The model has four actions: \texttt{SubmitOrder}, \texttt{CancelOrder},
\texttt{AmendOrder}, and \texttt{TimeAdvance}. \texttt{SubmitOrder}
nondeterministically selects all valid order parameters and invokes the full
processing pipeline. TLC's exhaustive exploration therefore considers every
valid combination of order type, side, price, quantity, TIF, STP policy,
iceberg flag, post-only flag, and minimum quantity, within the configured
bounds.


\subsection{Processing Pipeline}

The pipeline operator \texttt{ProcessOrder} implements all phases of the prose
specification's \texttt{PROCESS} procedure, including recursive operators for
the matching loop (\texttt{DoMatch}) and stop-trigger cascade
(\texttt{ProcessCascade}, \texttt{ProcessTriggeredStops}). Mutual recursion
between these operators is required to model cascading stop triggers correctly:
a triggered stop may generate trades, which may trigger further stops.

A \texttt{fuel} parameter on \texttt{DoMatch} bounds recursion depth for TLC
termination safety. Under the small quantity bounds used in model checking,
this bound is never reached in practice.

\subsection{What the Model Does Not Cover}

\begin{itemize}[nosep]
\item The model uses a single STP group identifier (\texttt{"G1"}). Multiple
      independent groups are not explored.
\item GTD expiry is modeled via \texttt{TimeAdvance} but without wall-clock
      constraints.
\item Event ordering (INV-10) and market-data publication are not modeled.
\item The model checks specification consistency, not implementation
      correctness.
\end{itemize}

\section{Findings and Counterexamples}
\label{sec:findings}
% ============================================================================

\subsection{Finding 1: FIFO Violation via Missing Timestamp on Triggered Stops}
\label{sec:bug1}

\paragraph{Severity.} High---directly violates INV-7/8.

\paragraph{Root cause location.} \S10.1 \texttt{EVALUATE\_STOPS} interaction
with \S6.1 \texttt{INSERT}.

\subsubsection*{The Mechanism}

The prose specification's stop-trigger procedure (\S10.1) converts a dormant
stop order to its base type (LIMIT or MARKET) and re-enters it into the
\texttt{PROCESS} pipeline. The stop order's \texttt{timestamp} field is not
updated at trigger time. It retains the timestamp assigned when the order was
originally submitted and placed into dormancy.

When the triggered LIMIT order does not fully fill and rests on the book via
\texttt{DISPOSE} (\S5.2), it is appended to the back of the appropriate
price-level queue per \S6.1---giving it the lowest priority at that level.
However, \texttt{FIFOWithinLevel} requires that the newly appended order have
a timestamp strictly greater than all orders already in the queue. Because
orders already in the queue arrived after the stop was submitted---they carry
later timestamps---the stop's original timestamp is numerically earlier. The
queue is now out of order.


\subsubsection*{Counterexample Trace}

TLC produced the following minimal counterexample. Times shown are logical
clock values.

\begin{figure}[H]
\centering
\begin{lstlisting}[language={},basicstyle=\ttfamily\footnotesize,frame=single,xleftmargin=0pt,framexleftmargin=0pt]
Step 1: Submit SELL LIMIT  qty=1, price=1      (id=1, ts=1)
        -> Rests on ask:  askQ[1] = [order1(ts=1)]

Step 2: Submit BUY STOP_LIMIT  stop=1, price=1, qty=1  (id=2, ts=2)
        -> lastTradePrice = NULL -> does not trigger
        -> Placed in stops = {order2(ts=2)}

Step 3: Submit BUY LIMIT  qty=2, price=1       (id=3, ts=3)
        -> Matches askQ[1]: fills order1 fully (fillQty=1)
        -> Trade: price=1, aggressor=3, passive=1
        -> lastTradePrice = 1
        -> order3: remainingQty=1, rests on bid
           bidQ[1] = [order3(ts=3)]
        -> Stop cascade: lastTradePrice(1) >= stopPrice(1)
           -> order2 triggers, converts to BUY LIMIT
           -> Timestamp NOT updated: ts=2 (STALE)
           -> Ask side empty, no fill
           -> Dispose: order2 rests on bid
              bidQ[1] = [order3(ts=3), order2(ts=2)]

        FIFOWithinLevel check:
            bidQ[1][1].timestamp = 3
            bidQ[1][2].timestamp = 2
            3 < 2 ?  FALSE  ->  INVARIANT VIOLATED
\end{lstlisting}
\caption{TLC counterexample trace for the FIFO violation. The triggered stop
order (id=2) retains its original timestamp (ts=2) and is appended behind an
order with a later timestamp (ts=3), producing a non-monotonic queue.}
\label{fig:trace}
\end{figure}

\subsubsection*{Why Manual Review Missed This}

The prose specification treats stop triggering and order insertion as separate
sections (\S10.1 and \S6.1 respectively). Each section is locally correct:
\S10.1 correctly describes conversion, and \S6.1 correctly describes
insertion. The interaction---that converted stops carry stale
timestamps---is invisible when reading either section in isolation. A
reviewer must simultaneously hold the conversion path, the insertion path, and
the FIFO invariant in mind to see the problem. TLC holds all three
simultaneously by construction.


\subsection{Finding 2: Iceberg Stranding Under STP DECREMENT}
\label{sec:bug2}

\paragraph{Severity.} Medium---creates a reachable book state with invalid
order geometry.

\paragraph{Root cause location.} \S8.3 DECREMENT case; missing interaction
with \S7.5 iceberg reload.

\subsubsection*{The Mechanism}

The STP DECREMENT policy reduces both the incoming order's
$\mathit{remainingQty}$ and the resting order's $\mathit{visibleQty}$ and
$\mathit{remainingQty}$ by
$\min(\mathit{incoming.remainingQty},\, \mathit{resting.visibleQty})$, without
generating a trade. The iceberg reload logic (\S7.5 step 3) specifies:

\begin{quote}
When $\mathit{visibleQty} = 0$ and $\mathit{remainingQty} > 0$, reload the
visible slice, assign a new timestamp, and move the order to the back of the
queue.
\end{quote}

However, this reload is only specified inside the fill path. The DECREMENT
action (\S8.3) does not reference \S7.5. There is no clause in \S8.3 for the
case where DECREMENT drives $\mathit{visibleQty}$ to zero while
$\mathit{remainingQty} > 0$.

The result is a resting iceberg order with $\mathit{visibleQty} = 0$ and
$\mathit{remainingQty} > 0$. It cannot be matched---$\mathit{fillQty} =
\min(\mathit{incoming.remainingQty}, 0) = 0$---and cannot reload because the
reload trigger is only in the fill path. The order is stranded: it occupies a
queue position, blocks no fills, and cannot be removed except by explicit
cancel.

\subsubsection*{Illustrative Scenario}

\begin{lstlisting}[language={},basicstyle=\ttfamily\footnotesize,frame=single,xleftmargin=0pt,framexleftmargin=0pt]
Resting iceberg: qty=2, displayQty=1, visibleQty=1,
                 remainingQty=2, stpGroup="G1"        (id=R)
Incoming:        qty=1, stpGroup="G1", stpPolicy=DECREMENT  (id=I)

DECREMENT:
    reduceQty = min(1, 1) = 1
    I.remainingQty = 0   -> I is cancelled
    R.visibleQty   = 0
    R.remainingQty = 1   -> R remains on book

Post-decrement state of R:
    visibleQty=0, remainingQty=1, displayQty=1
    -> Reload path not invoked
    -> R sits on book, invisible, unreachable by matching
\end{lstlisting}

\paragraph{Note on claim level.} This scenario is reachable in the prose
specification as written. However, the TLA+ model as checked incorporates the
fix (iceberg reload after DECREMENT) because the stranded state would cause
the model checker itself to loop: subsequent orders would find
$\mathit{visibleQty} = 0$ and skip the order without decrementing fuel,
creating an effective infinite loop in \texttt{DoMatch}. The bug is therefore
presented as a specification gap---a path the spec silently omits---rather
than as a TLC-witnessed invariant violation with a precise counterexample
trace.


\subsection{Design Observations (Not Demonstrated Bugs)}

The following interactions are noted as potential specification ambiguities.
They are not presented as verified bugs because no invariant violation was
produced for them.

\paragraph{Observation 1---FOK + STP DECREMENT.}
The FOK pre-check counts non-conflicting visible quantity. During matching,
DECREMENT can consume the incoming order's $\mathit{remainingQty}$ without
generating trades. The incoming order reaches $\mathit{remainingQty} = 0$ and
is treated as cancelled, but with fewer trade fills than $\mathit{qty}$.
Whether this constitutes a FOK violation depends on whether FOK is defined as
``fully traded'' or ``$\mathit{remainingQty}$ fully consumed.'' The spec does
not address this interaction.

\paragraph{Observation 2---MinQty + STP DECREMENT.}
Analogously, the MinQty pre-check guarantees
$\mathit{filled\_quantity} \geq \mathit{minQty}$ (\S5.4), but if DECREMENT
consumes part of the incoming quantity before any fills occur, the traded
quantity may fall below $\mathit{minQty}$ even though the pre-check passed.

\paragraph{Observation 3---MTL + MinQty clearing.}
The spec clears $\mathit{minQty}$ in two places: once in the MTL path (Phase
4) and once in the normal matching path (Phase 5a). This duplication is
currently correct but fragile---adding a new order type or routing path
requires remembering to add a third clearing point.

\section{Specification Fixes}
\label{sec:fixes}
% ============================================================================

Both confirmed issues were fixed in the TLA+ model. The fixes are minimal and
directly address the root causes.

\begin{table}[H]
\centering
\caption{Bug and fix summary.}
\label{tab:fixes}
\small
\begin{tabular}{@{}lL{3cm}L{3.5cm}L{4.5cm}l@{}}
\toprule
ID & Location & Description & Fix & Invariant \\
\midrule
BUG-1 & \S10.1 \texttt{EVALUATE\_STOPS}
      & Triggered stop retains original timestamp; violates FIFO when resting
      & Assign \texttt{stop.timestamp = currentTimestamp()} before re-entering pipeline
      & INV-7/8 \\
\addlinespace
BUG-2 & \S8.3 DECREMENT
      & Iceberg $\mathit{visibleQty}$ driven to 0 without reload; order stranded
      & After DECREMENT: if $\mathit{visibleQty} = 0 \land \mathit{remainingQty} > 0 \land \mathit{displayQty} \neq \bot$, reload visible slice, refresh timestamp, move to back
      & Spec gap \\
\bottomrule
\end{tabular}
\end{table}


\subsection{Fix for BUG-1 (Stop Timestamp)}

The fix is a one-line addition to \S10.1. In the TLA+ model,
\texttt{ProcessTriggeredStops} applies the fix:

\begin{lstlisting}[language=tlaplus]
converted == [ConvertStop(s) EXCEPT !.timestamp = tm]
\end{lstlisting}

The corresponding prose amendment to \S10.1:

\begin{lstlisting}[language={}]
-- After "Convert to base type":
stop.timestamp = currentTimestamp()   -- Assign trigger-time priority
PROCESS(stop, book)
\end{lstlisting}

This ensures that a triggered stop order receives a timestamp reflecting when
it entered the live matching pipeline, not when it was originally submitted.


\subsection{Fix for BUG-2 (Iceberg Reload After DECREMENT)}

The fix adds a post-DECREMENT iceberg check to \S8.3. In the TLA+ model:

\begin{lstlisting}[language=tlaplus]
ELSE IF newRest.visibleQty = 0 /\ newRest.displayQty /= NULL THEN
    LET reloadQty == Min(newRest.displayQty, newRest.remainingQty)
        reloaded  == [newRest EXCEPT !.visibleQty = reloadQty,
                                     !.timestamp = tm,
                                     !.status = "PARTIAL"]
        sc == SetContra(side, bQ, aQ, bestP,
                        restTail \o <<reloaded>>)
    IN DoMatch(newInc, sc.bQ, sc.aQ, tds, tm + 1, fuel - 1)
\end{lstlisting}

The corresponding prose amendment to \S8.3, after the DECREMENT quantity
update:

\begin{lstlisting}[language={}]
IF resting.visibleQty = 0 AND resting.remainingQty > 0
   AND resting.displayQty != null:
    resting.visibleQty = min(resting.displayQty, resting.remainingQty)
    resting.timestamp  = currentTimestamp()
    MOVE resting TO back OF level.orders
\end{lstlisting}

This mirrors the iceberg reload behavior specified in \S7.5 for the normal
fill path, extending it to the DECREMENT path for consistency.

\section{Implementation via Guided Synthesis}
\label{sec:synthesis}
% ============================================================================

A unique aspect of this case study is the direct translation of the fixed 
formal specification into high-performance C++ and C implementations. 

\subsection{The ``One-Shot'' Methodology}

Rather than traditional manual coding, we employed a paradigm of 
\textbf{Guided Synthesis}. The fixed formal specification (prose + TLA+ fixes) 
was provided to a Large Language Model (Claude 3.5 Sonnet) as the primary 
``logical prompt.'' We augmented this with two architectural hints:
\begin{enumerate}
    \item Use of \textbf{Intrusive Doubly-Linked Lists} for $O(1)$ order 
          removal and FIFO management.
    \item Use of \textbf{TLSF (Two-Level Segregate Fit)} pool allocation 
          to eliminate heap fragmentation and ensure deterministic performance.
\end{enumerate}

The LLM synthesized the implementations in a single pass. The resulting code 
was not only structurally optimized (e.g., using cache-line-aware struct packing 
in C) but also logically correct relative to the formal model.

\subsection{Mechanical Conformance}

The synthesized implementations correctly applied the subtle fixes identified 
by TLA+:
\begin{itemize}
    \item \textbf{C Implementation}: In \texttt{evaluate\_stops}, the engine 
          assigns \texttt{stop->m\_timestamp = e->m_clock++} before re-entry, 
          preserving FIFO.
    \item \textbf{C++ Implementation}: In the STP handler, the engine reloads 
          the \texttt{visibleQty} and refreshes the timestamp if a 
          \texttt{DECREMENT} zeros the display quantity.
\end{itemize}

Furthermore, the C implementation includes an \texttt{engine\_check\_invariants} 
function that translates the TLA+ safety properties into runtime \texttt{assert()} 
statements. This creates a ``Continuous Verification'' loop where the production 
code is constantly checked against the formal model's logic.

\subsection{Bridging the Implementation Gap}

This result suggests that the ``Implementation Gap'' is closing. When a 
specification is mathematically rigorous (TLA+) and structurally sound 
(Lean 4), it serves as an unambiguous blueprint. An AI assistant can then 
act as a ``high-level compiler,'' translating those logical truths into 
optimized, low-level code without the typical ``trial-and-error'' bug cycles.

\section{Lessons Learned}
\label{sec:lessons}
% ============================================================================

\subsection{Feature Composition Is Where Bugs Hide}

Both findings share a structural pattern: each involved feature is correctly
specified in isolation, but the specification omits a clause covering their
interaction. Stop orders are correctly described; iceberg reload is correctly
described; FIFO insertion is correctly described. None of these sections
contains an error when read alone. The bug lives in the intersection: what
happens to an iceberg order's timestamp when the mechanism that zeros its
visible quantity is not the fill path but the STP DECREMENT path?

This is not a failure of careful writing---it is a structural limitation of
prose specifications. Prose sections have natural scope boundaries.
Interactions between features that span sections are systematically harder to
see than behaviors within a single section.

A formal model collapses all features into a single operational semantics and
evaluates invariants globally. The intersection cases are not special---they
are just more states.


\subsection{Logical Clocks Amplify Ordering Bugs}

The FIFO invariant is stated in terms of timestamps. Because the
specification uses a monotonically increasing logical clock, any state
transition that fails to advance the clock before inserting an order can
produce a stale timestamp that violates ordering. The stop-trigger bug is
exactly this: the clock was not advanced before the triggered stop was assigned
to the book.

Timestamp-based ordering invariants are fragile in the presence of features
(like stop triggers) that re-enter orders into the pipeline from a dormant
pool. Wherever an order exits a dormant state and re-enters an active queue,
its timestamp must be refreshed. The fix is simple; the risk is that prose
specifications do not enumerate every re-entry point.


\subsection{Iceberg Reload Is Not Tied Tightly Enough to Its Trigger Condition}

The iceberg reload semantics in \S7.5 are described in terms of the fill path:
``when $\mathit{visibleQty}$ is fully consumed [by a fill].'' The prose
implicitly assumes that $\mathit{visibleQty}$ can only reach zero through the
fill path. STP DECREMENT provides a second path to zero
$\mathit{visibleQty}$ that the reload trigger does not anticipate.

A more robust specification would state the reload condition declaratively as
an invariant on the resting order state:

\begin{quote}
A resting order must not have $\mathit{visibleQty} = 0$ and
$\mathit{remainingQty} > 0$.
\end{quote}

\noindent Rather than specifying reload as a step triggered by the fill path,
it would be specified as a consequence of this invariant: any action that could
produce this state must be followed by a reload. This declarative framing
would make the omission in DECREMENT immediately apparent.


\subsection{Bounded Exploration Is Sufficient for Shallow Bugs}

Both bugs in this report were found in the 2-order, 2-quantity, 2-price
configuration (11.5 million states, 102 seconds). They did not require large
configurations. Shallow bugs---those reachable within a small number of
orders---are precisely the category that bounded exhaustive exploration handles
well. The two-order FIFO counterexample requires exactly three orders (one
resting ask, one stop, one incoming buyer) and is found quickly.


\subsection{The Specification Artifact Has Independent Value}

The TLA+ model serves as a precise, executable reference for the prose
specification. Independently of the bugs found, it forces every semantic
choice to be made explicit: what is the recursion structure of stop cascades?
What exactly does ``timestamp'' mean for a triggered order? At what point
does MTL price derivation occur relative to the cascade? These questions have
definite answers in the model even when the prose is ambiguous.

\section{Limitations and Reproducibility}
\label{sec:limitations}
% ============================================================================

\subsection{Bounded Exhaustive Exploration, Not Proof}

TLC performs bounded exhaustive state enumeration: it finds every state
reachable from \texttt{Init} via the \texttt{Next} relation, within the
parameter bounds. A clean run means no invariant violation was found within
the explored configurations. It does not mean no violation exists for larger
configurations, more orders, or more price levels. We do not claim the
specification is correct in any unbounded sense.

Specifically:
\begin{itemize}[nosep]
\item The 3-order partial run did not complete state-space exploration.
\item No configuration with more than two distinct STP groups was checked.
\item GTD time-based expiry was checked only via the simplified
      \texttt{TimeAdvance} action, not against wall-clock constraints.
\item The event-ordering invariant (INV-10) is not formalized in the model.
\end{itemize}


\subsection{Specification vs.\ Implementation}

This work verifies the prose specification against itself. The TLA+ model is
a formalization of the prose spec; invariant violations indicate
inconsistencies or omissions in the spec. An implementation that correctly
implements the (fixed) specification could still contain implementation
bugs---off-by-one errors, concurrency hazards, data-structure
corruption---that are outside the scope of this model.


\subsection{Reproducibility}

The complete artifacts are:
\begin{itemize}[nosep]
\item \texttt{MatchingEngine.tla}---the TLA+ specification ($\sim$850 lines)
\item \texttt{MatchingEngine.cfg}---TLC configuration (default)
\item \texttt{MatchingEngine\_*.cfg}---per-configuration files for Table~\ref{tab:configs}
\item \texttt{matching-engine-formal-spec.md} v1.2.0---the prose specification
\item \texttt{REPORT.md}---model checking output and bug descriptions
\item \textbf{Synthesized C/C++ implementations}---available in the repository.
\end{itemize}

To reproduce the FIFO counterexample:
\begin{enumerate}[nosep]
\item Revert the stop timestamp fix in \texttt{ProcessTriggeredStops}: replace\\
      \texttt{[ConvertStop(s) EXCEPT !.timestamp = tm]} with
      \texttt{ConvertStop(s)}.
\item Run TLC with the Small configuration
      (\texttt{MAX\_ORDERS=2, MAX\_QTY=2, PRICES=\{1,2\}}, amend disabled).
\item TLC will find a violation of \texttt{FIFOWithinLevel} within the first
      several million states.
\end{enumerate}

\section{Conclusion}
\label{sec:conclusion}
% ============================================================================

We formalized a 900-line prose matching-engine specification in TLA+ and ran
bounded exhaustive exploration across several configurations totaling over 36
million distinct states. Within the explored configurations, TLC found one
definitive specification bug---a FIFO ordering violation caused by a triggered
stop order retaining its original submission timestamp---and surfaced one
specification gap---the iceberg reload path is not invoked after STP DECREMENT
zeros a resting iceberg's visible quantity.

The flagship finding is the FIFO violation. Its counterexample is a
three-order sequence, operationally simple, not obviously related to stop
orders when reading the FIFO invariant definition. Normal prose review of the
specification did not find it. TLC found it in under two minutes.

More importantly, the resulting ``verified spec'' enabled the one-shot 
synthesis of high-performance implementations that were correct-by-construction. 
The broader lesson is that formal methods and AI-assisted synthesis are 
complementary: formal methods provide the \textbf{truth}, and AI provides 
the \textbf{performance}, finally bridging the gap from prose to production.
